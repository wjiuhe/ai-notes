Step-by-step walkthrough: Build the same expense tracker with each framework

author: Jiuhe Wang | published: February 23, 2026 | difficulty: BEGINNER-INTERMEDIATE

Standard Feature Request

Fair Evaluation: To compare frameworks objectively, we'll use the exact same feature request with each framework. Only the workflow and tools differ—not the requirements.
Standardized Input (Use This for All Frameworks)
Feature Request: Personal Expense Tracker API
Build a RESTful API for tracking personal expenses with the following
requirements:
Tech Stack:
- FastAPI (Python) for the backend
- PostgreSQL for persistent storage
- SQLAlchemy for ORM
- Pydantic for data validation
- pytest for testing
Core Features:
1. Create new expense (amount, category, description, date)
2. Read/List expenses with pagination
3. Update existing expense
4. Delete expense
5. Filter expenses by category and date range
6. Get monthly spending summary by category
Validation Requirements:
- Amount must be positive (> 0)
- Category must be from allowed list: Food, Transport, Entertainment,
Utilities, Healthcare, Shopping, Other
- Description is optional, max 255 characters
- Date format: ISO 8601 (YYYY-MM-DD)
Testing Requirements:
- Unit tests for models and validation
- Integration tests for API endpoints
- Minimum 80% code coverage

Project Overview

What We're Building

Personal Expense Tracker API

A RESTful API for tracking personal expenses with the following features:

  • Create, read, update, delete expenses
  • Filter expenses by category and date range
  • Get spending summary reports
  • Validate expense data (positive amounts, valid categories)
  • Persistent storage with PostgreSQL

Tech Stack:

  • Backend: FastAPI (Python)
  • Database: PostgreSQL
  • Testing: pytest with coverage
  • Validation: Pydantic models
  • Migrations: Alembic

Everything Claude Code Walkthrough

ECC Approach
"Agent-driven development with comprehensive tooling"
ECC provides specialized agents and commands that guide you through the entire development process. We'll use the /plan agent for design, /tdd for test-driven development, and built-in hooks for quality enforcement.

Step 1: Project Setup

1Initialize Project
Terminal
# Create project directory
mkdir expense-tracker-ecc
cd expense-tracker-ecc
# Initialize git
git init
# Create virtual environment
python -m venv venv
source venv/bin/activate # On Windows: venv\Scripts\activate
# Install ECC plugin
/plugin marketplace add affaan-m/everything-claude-code
/plugin install everything-claude-code@everything-claude-code

Step 2: Create Design Document

2Use /plan Agent

In Claude Code, run the planning command:

Claude Code Command
/plan

When prompted, paste the Standard Feature Request from the section above. ECC will create a comprehensive design document with:

  • API endpoint specifications
  • Database schema design
  • Project structure recommendations
  • Testing strategy
  • Implementation phases

Step 3: Implement with /tdd

3Test-Driven Development

ECC's TDD agent will guide you through writing tests first:

Start TDD Workflow
/tdd

The agent will:

  1. Ask which feature to implement first
  2. Create a failing test
  3. Guide you to write minimal code to pass
  4. Run tests automatically
  5. Suggest refactoring

Example workflow:

$ /tdd
[ECC TDD Agent] What feature shall we implement?
> Create expense model with validation
[ECC] Creating test for Expense model validation...
✓ Test created: tests/test_models.py::test_expense_validation
✓ Test fails as expected (RED phase)
[ECC] Now implement the Expense model. Requirements:
- Amount must be positive
- Category must be from allowed list
- Description required, max 255 chars

Step 4: Database Setup

4Configure PostgreSQL
Database Configuration
# Create .env file
DATABASE_URL=postgresql://postgres:password@localhost:5432/expenses
# Run ECC database setup command
/setup database
# This creates:
# - SQLAlchemy models
# - Alembic migrations
# - Database connection pool

Step 5: API Implementation

5Create API Endpoints

Use the /implement command for guided implementation:

ECC Commands
# Implement each endpoint
/implement "Create POST /expenses endpoint"
/implement "Create GET /expenses endpoint with filters"
/implement "Create PUT /expenses/{id} endpoint"
/implement "Create DELETE /expenses/{id} endpoint"
/implement "Create GET /expenses/summary endpoint"

Step 6: Quality Assurance

6Run Comprehensive Checks
Quality Commands
# Run all tests
/test
# Run with coverage
/test --coverage
# Full verification (tests, lint, type-check, security)
/verify
# Security scan
/security-scan
ECC Complete! You now have a fully functional expense tracker built from the Standard Feature Request with tests, documentation, and quality checks. ECC's agents guided you through each step while enforcing best practices.

BMAD Method Walkthrough

BMAD Approach
"Structured agile process with formal planning"
BMAD takes you through a complete agile process: analysis, planning, solutioning, and implementation. We'll create formal PRDs and user stories before writing any code.

Step 1: Install BMAD

1Setup BMAD Method
Terminal
# Create project directory
mkdir expense-tracker-bmad
cd expense-tracker-bmad
# Initialize project
git init
python -m venv venv
source venv/bin/activate
# Install BMAD
npx bmad-method install
# Select: BMad Method (BMM) when prompted

Step 2: BMad-Help Guidance

2Get Project Guidance

Use BMad-Help to get context-aware guidance:

BMAD Command
/bmad-help

Paste the Standard Feature Request from the section above. BMad-Help will analyze your project state and recommend:

  • Which BMAD phase to start with
  • Recommended workflow (Quick Flow, BMad Method, or Enterprise)
  • Next steps specific to your project

Step 3: Phase 1 - Analysis

3Create Analysis Documents
Analysis Phase
# Start analysis phase
/bmad-bmm-analysis
# BMAD creates:
# - docs/analysis/initial-brainstorming.md
# - docs/analysis/research-findings.md
# - docs/analysis/user-personas.md

Review and refine the analysis documents with the BMAD agent.

Step 4: Phase 2 - Planning

4Create PRD

Generate a formal Product Requirements Document:

Planning Phase
# Create Product Requirements Document
/bmad-bmm-create-prd
# Review PRD interactively
/bmad-bmm-review-prd

The PRD will include:

  • Executive summary
  • User stories and use cases
  • Functional requirements
  • Non-functional requirements
  • API specifications
  • Database schema

Step 5: Phase 3 - Solutioning

5Architecture Design
Solutioning Phase
# Create architecture document
/bmad-bmm-architecture
# Select architecture type:
# - Quick Flow (1-15 stories)
# - BMad Method (10-50+ stories)
# - Enterprise (30+ stories, complex)

For our expense tracker, choose Quick Flow or BMad Method.

Step 6: Phase 4 - Implementation

6Story-Driven Development

BMAD breaks implementation into user stories:

Create Stories
# Create story from PRD
/bmad-bmm-create-story
# Example: "As a user, I want to create an expense"

For each story:

  1. Create story file with acceptance criteria
  2. Run /bmad-bmm-dev-story to implement
  3. Write tests as part of story
  4. Run /bmad-bmm-test-story to verify
  5. Run /bmad-bmm-code-review for quality check
  6. Mark story complete

Example story workflow:

$ /bmad-bmm-create-story
Story: Create expense with validation
Acceptance Criteria:
[ ] User can POST /expenses with amount, category, description
[ ] Amount must be positive number
[ ] Category from allowed list
[ ] Returns 201 on success with expense ID
$ /bmad-bmm-dev-story story-001.md
[BMAD] Implementing story: Create expense...
[BMAD] Writing tests first...
[BMAD] Implementing endpoint...
[BMAD] Running tests...
✓ All tests passing
BMAD Complete! You now have a fully documented expense tracker built from the Standard Feature Request with PRD, architecture docs, user stories, and tested implementation following formal agile processes.

Claude Pilot Walkthrough

Pilot Approach
"Quality enforced through automated hooks"
Claude Pilot uses hooks to enforce quality automatically. The TDD enforcer won't let you commit code without tests, and the smart model routing uses Opus for planning while Sonnet handles implementation.

Step 1: Install Pilot

1Setup Claude Pilot
Terminal
# Install Pilot
curl -fsSL https://raw.githubusercontent.com/maxritter/claude-pilot/main/install.sh | bash
# Start Pilot in new project
mkdir expense-tracker-pilot
cd expense-tracker-pilot
pilot

Step 2: Sync Project

2Initialize and Sync

Pilot needs to understand your project structure:

Pilot Commands
# Initialize git and Python
git init
python -m venv venv
source venv/bin/activate
# Sync project with Pilot
/sync

The sync command:

  • Explores your directory structure
  • Detects tech stack (FastAPI, PostgreSQL, pytest)
  • Creates project.md with project context
  • Configures hooks for your environment

Step 3: Create Specification

3Spec-Driven Development

Pilot uses specifications that must be approved before implementation:

Create Specification
/spec

Paste the Standard Feature Request from the section above. Pilot will create a detailed spec.md with:

  • Requirements breakdown
  • Implementation approach
  • Test scenarios
  • Validation checklist
$ /spec
[Pilot] Analyzing requirements...
[Pilot] Routing to Opus for planning...
Created spec.md with:
- 5 API endpoints specified
- Database schema designed
- 12 test scenarios defined
Review spec.md and run /approve to continue

Step 4: Approve and Implement

4Approve Specification
Implementation
# Review spec.md, then approve
/approve
# Start implementation
/implement

Pilot's implementation process:

  1. Routes planning to Opus (deep reasoning)
  2. Routes implementation to Sonnet (fast execution)
  3. PostToolUse hooks run after each file change:
    • Runs lint/format checks
    • TDD enforcer checks for tests
    • Runs affected tests
  4. Routes verification to Opus (quality review)

Step 5: Work with Hooks

5Hook-Enforced Quality

Pilot's hooks work automatically. Here's what happens when you modify files:

$ Modified: app/models.py
[Pilot Hook: PostToolUse]
→ File checker: Running ruff...
→ TDD enforcer: Checking for tests...
⚠ WARNING: No tests found for app/models.py
✗ BLOCKED: Create tests before proceeding
$ Created: tests/test_models.py
[Pilot Hook: PostToolUse]
→ File checker: ✓
→ TDD enforcer: ✓ Tests found
→ Running affected tests...
✓ All tests passing

Step 6: Independent Verification

6Verify with Opus
Quality Verification
# Independent verification (always uses Opus)
/verify
# Try to commit (blocked if issues)
/commit "Add expense CRUD endpoints"

The /commit command:

  • Runs Stop hook validation
  • Checks all tests pass
  • Verifies test coverage threshold
  • Ensures spec requirements met
  • Only then allows commit
Pilot Complete! You have a production-ready expense tracker built from the Standard Feature Request with enforced TDD, automated quality checks, and smart model routing. The hooks prevented any code without tests from being committed.

Superpowers Walkthrough

Superpowers Approach
"Rigid TDD discipline with the Iron Law"
Superpowers enforces strict TDD through its skill system. Code without failing tests is deleted. The workflow is auto-triggered and uses two-stage reviews for every task.

Step 1: Install Superpowers

1Setup Superpowers
Terminal
# Create project
mkdir expense-tracker-superpowers
cd expense-tracker-superpowers
# Initialize
git init
python -m venv venv
source venv/bin/activate
# Install Superpowers
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers@superpowers-marketplace

Step 2: Trigger Auto-Workflow

2Just Describe Your Goal

Superpowers skills auto-trigger based on your input. Paste the Standard Feature Request from the section above.

Superpowers automatically activates skills in sequence:

[Skill: brainstorming] Activated
Socratic design refinement starting...
Q: What categories should be allowed for expenses?
> Food, Transport, Entertainment, Utilities, Other
Q: Should users be authenticated?
> Not for this MVP
[Skill: using-git-worktrees] Activated
Creating feature branch: feature/expense-tracker
[Skill: writing-plans] Activated
Breaking work into 8 bite-sized tasks...

Step 3: The Iron Law of TDD

3RED-GREEN-REFACTOR

"NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"

Write code before the test? Delete it. Start over. No exceptions.

Superpowers enforces this strictly:

[Superpowers] Task: Create Expense model
RED PHASE:
✓ Writing failing test
✓ Watching test FAIL
GREEN PHASE:
✓ Writing minimal code to pass
✓ Watching test PASS
REFACTOR PHASE:
→ Improving code quality...
✓ Tests still passing
[Two-Stage Review]
Stage 1: Spec Compliance ✓
Stage 2: Code Quality ✓
Task complete. Moving to next task...

Step 4: Two-Stage Review

4Every Task Gets Reviewed

Superpowers dispatches fresh subagents for reviews:

Review Process
# Stage 1: Spec Compliance Review
- Does it meet requirements?
- Are edge cases handled?
- Is the test coverage adequate?
# Critical? → Block and fix
# Stage 2: Code Quality Review
- Code style and readability
- Design patterns used
- Performance considerations
# Then: Next Task

Step 5: Subagent-Driven Development

5Each Task Gets a Fresh Agent

Superpowers breaks work into 2-5 minute tasks and dispatches subagents:

Example Task Breakdown
Task 1: Create database models (3 min)
Task 2: Create Pydantic schemas (2 min)
Task 3: Implement CRUD operations (5 min)
Task 4: Create POST endpoint (3 min)
Task 5: Create GET endpoints (4 min)
Task 6: Add filtering support (4 min)
Task 7: Create summary endpoint (3 min)
Task 8: Final verification (2 min)

Each task:

  • Gets a fresh subagent (clean context)
  • Follows TDD (must see test fail first)
  • Goes through two-stage review
  • Commits after REFACTOR phase

Expense tracker implementation complete!
Superpowers Complete! You have a rigorously tested expense tracker built with strict TDD discipline using the exact same Standard Feature Request as the other frameworks. Every line of code has a corresponding test, and the two-stage review process ensured quality at each step.

Comparing the Four Approaches

Fair Comparison: Each framework received the exact same Standard Feature Request shown at the top of this guide. The only differences are the workflows, commands, and enforcement mechanisms each framework provides.

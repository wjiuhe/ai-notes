AI Frameworks Testing Deep Dive

Production-grade testing workflows for every framework

author: Jiuhe Wang | published: February 23, 2026 | difficulty: INTERMEDIATE

Testing Philosophy

Testing is where AI-assisted development falls apart—or comes together. Each framework takes a different path to the same destination: code that actually works in production.

"Here's the thing about tests: they're not documentation. They're executable specs. If they pass, your code does what it should. If they fail, it doesn't."

— The Practical View

Framework Testing Philosophies

Everything Claude Code
"Agent-driven testing"
Uses /test, /tdd, /e2e commands with Eval Harness for evaluation. Testing lives inside the agent ecosystem. Good if you want everything in one place.
BMAD Method
"Story-driven acceptance testing"
Tests get written as part of story development. You build the story, you build the test. Uses structured agile practices.
Superpowers
"Iron Law: Tests first"
Rigid TDD. Code without failing tests gets deleted. Two-stage review: spec compliance first, then code quality. No exceptions.

The Testing Pyramid

E2E Tests (5%) Slow, expensive, critical paths
↓
Integration Tests (15%) Component interactions, APIs
↓
Unit Tests (80%) Fast, cheap, business logic
Cost/Speed: Unit < Integration < E2E
Confidence: E2E > Integration > Unit

Test Types Explained

Unit Tests

Test individual functions, methods, or classes in isolation. Fast, deterministic, and should cover edge cases and business logic.

Characteristics:

  • Run in milliseconds
  • No external dependencies (mocked)
  • Test one thing at a time
  • High coverage target: 80%+
Python Example
# test_calculator.py
import pytest
from calculator import add, divide
def test_add_positive_numbers():
    assert add(2, 3) == 5
def test_add_negative_numbers():
    assert add(-2, -3) == -5
def test_divide_by_zero_raises():
    with pytest.raises(ZeroDivisionError):
        divide(10, 0)
Integration Tests

Test how components work together. Verify database connections, API endpoints, service interactions, and external integrations.

Characteristics:

  • Run in seconds
  • Use real dependencies (test DB)
  • Test component interactions
  • Medium coverage target: 60%+
FastAPI Example
# test_api.py
from fastapi.testclient import TestClient
from main import app
client = TestClient(app)
def test_create_expense():
    response = client.post("/expenses/", json={
        "amount": 100.00,
        "category": "food",
        "description": "Lunch"
    })
    assert response.status_code == 201
    data = response.json()
    assert data["amount"] == 100.00
    assert "id" in data
E2E Tests

Test complete user workflows from start to finish. Simulate real user interactions through the full application stack.

Characteristics:

  • Run in minutes
  • Full stack with real browser/HTTP
  • Test critical user paths
  • Low coverage target: 20%+
Playwright Example
# test_e2e.py
from playwright.sync_api import Page, expect
def test_user_creates_expense(page: Page):
    # Navigate to app
    page.goto("http://localhost:8000")
    
    # Fill form
    page.fill("[name='amount']", "100.00")
    page.select_option("[name='category']", "food")
    page.fill("[name='description']", "Lunch")
    
    # Submit and verify
    page.click("button[type='submit']")
    expect(page.locator(".success-message")).to_be_visible()

Everything Claude Code: Testing Deep Dive

ECC Testing Approach
"Agent-driven testing with comprehensive coverage"
ECC provides specialized testing agents and commands that integrate testing throughout the development lifecycle. The Eval Harness provides systematic test evaluation and reporting.

Testing Commands

/tdd
Start Test-Driven Development workflow. Creates failing test first, then implements code to pass.
/test
Run test suite with smart filtering. Supports running specific test files, patterns, or coverage reports.
/e2e
Execute end-to-end tests with browser automation. Includes screenshot capture on failure.
/verify
Comprehensive verification: tests, lint, type-check, and security scan in one command.

ECC Testing Best Practices

  • Use /tdd for new features: Always start with failing tests
  • Leverage test agents: Let /test agent suggest edge cases
  • Run /verify before commits: Catch issues before they reach CI
  • Organize tests properly: ECC enforces tests/unit/, tests/integration/, tests/e2e/ structure
  • Use fixtures: ECC generates fixtures for common test data patterns
ECC Testing Tip: The Eval Harness provides systematic test evaluation. Run /eval to get a comprehensive report on test quality, coverage gaps, and suggestions for improvement.

Superpowers: Testing Deep Dive

Superpowers Testing Approach
"Rigid TDD discipline with the Iron Law"
Superpowers enforces TDD through its skill system. The test-driven-development skill auto-triggers and ensures no production code exists without a failing test first. Violations result in code deletion.

The Iron Law of TDD

"NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"

Write code before the test? Delete it. Start over.
Delete means delete. No exceptions.
No "I'll add tests later."
No "This is just temporary."

— Superpowers TDD Skill

Superpowers TDD Workflow

RED PHASE Write failing test Watch it FAIL (runtime)
↓
GREEN PHASE Write minimal code to pass Watch it PASS
↓
REFACTOR PHASE Improve code quality Keep tests passing → Commit

Superpowers Testing Skills

test-driven-development (auto-trigger)
Core skill that enforces the Iron Law. Activates when you mention writing code or tests.
verification-before-completion
Requires running tests and verifying output before claiming completion.
subagent-driven-development
Dispatches subagents with TDD requirements for each task.
receiving-code-review
Two-stage review: spec compliance first, then code quality including test coverage.

Debugging Failed Tests

Common Test Failures and Solutions

Assertion Failures

Test logic is correct but expected value doesn't match.

FAILED test_expense.py::test_calculate_total
AssertionError: assert 150.0 == 100.0

Debug Steps:

  1. Add print/logging to see actual values
  2. Check if test is testing the right thing
  3. Verify implementation matches requirements
  4. Look for floating point precision issues
Timeout Failures

Test takes too long, often due to external dependencies.

FAILED test_api.py::test_slow_endpoint - Timeout > 30s

Solutions:

  • Mock external services
  • Use test database, not production
  • Mark slow tests with @pytest.mark.slow
  • Run with --timeout flag
Database State Issues

Tests interfere with each other due to shared database state.

FAILED test_user.py::test_create_user - UniqueViolation: email already exists

Solutions:

  • Use pytest fixtures with transaction rollback
  • Generate unique test data (UUIDs, timestamps)
  • Reset database state between tests
  • Use factories, not hardcoded data

Coverage Configuration

.coveragerc
[run]
source = src
omit =
*/tests/*
*/venv/*
*/migrations/*
src/__main__.py
[report]
exclude_lines =
pragma: no cover

Coverage vs Quality: High coverage doesn't guarantee good tests. A test that runs code but doesn't assert anything gives 100% coverage but 0% confidence. Focus on meaningful assertions, not just coverage numbers.

Testing Best Practices

Universal Best Practices

  • Test behavior, not implementation: Test what code does, not how it does it
  • One assertion per test: Each test should verify one thing
  • Use descriptive names: test_calculate_total_with_multiple_items not test_calc
  • Arrange-Act-Assert: Structure tests clearly
  • Independent tests: Tests should not depend on each other
  • Fast tests: Unit tests should run in milliseconds
  • Deterministic tests: Same input → same output, always

Test Smells to Avoid

Smell Problem Solution
Mystery Guest Test uses external data not visible in test Use fixtures with explicit setup
Assertion Roulette Multiple assertions, unclear which failed Split into separate tests
Happy Path Only Only tests ideal scenarios Test edge cases and errors
Sleepy Tests Uses time.sleep() for synchronization Use proper async/await or polling
Golden Master Large snapshot tests that are hard to review Use targeted assertions
